NiCo – Hilfe

NiCo ist ein lokaler, browserbasierter Konfigurator für NixOS. Er arbeitet direkt mit echten .nix-Dateien auf 127.0.0.1 – ohne Telemetrie, ohne Daten nach außen, ohne eigene Konfig-Welt. NiCo soll Nix nicht ersetzen, sondern die Arbeit damit übersichtlicher und sicherer machen.

1Das Herzstück: Nix-Brix

1.1Was sind Nix-Brix?

Nix-Brix sind frei editierbare Nix-Abschnitte, die NiCo niemals verändert. Wer Konfiguration braucht, die NiCo nicht als Formularfeld kennt – ein Dienst, ein Overlay, eine spezielle Option – schreibt sie als Nix-Brick. NiCo umschließt den Brick mit Marker-Kommentaren wie # <brick: End / #1 syncthing> … # </brick: syncthing> und lässt den Inhalt dazwischen vollständig unberührt.

Das Konzept ist in dieser Form einzigartig: Statt nur das zu erlauben, was ein GUI-Tool vorgeneriert, bleibt die volle Nix-Freiheit erhalten. Nix-Brix und Formularfelder koexistieren in derselben Datei – NiCo kümmert sich nur um seinen eigenen Bereich.

Wichtig: Nix-Brix werden beim Import einer bestehenden Konfiguration automatisch angelegt – unbekannte Abschnitte landen als Brick 1-imported-config und gehen nicht verloren.

1.2Brix anlegen und verwalten

Einen neuen Nix-Brick anlegen: Schaltfläche + Nix-Brick einfügen → Bezeichnung eingeben → Einfügen. Der Brick erscheint am Ende des Dokuments; der Inhalt wird direkt im Codefenster rechts bearbeitet.

Über das Aktionsmenü (⋮) eines Bricks stehen weitere Funktionen bereit:

  • Umbenennen – ändert die Bezeichnung, die numerische ID bleibt erhalten
  • Verschieben – legt den Brick in einen anderen Abschnitt oder an eine andere Position
  • Aufteilen – teilt einen Brick an einer bestimmten Zeile in zwei separate Brix
  • Löschen – entfernt den Brick unwiderruflich; der Inhalt geht verloren

Nix-Brix lassen sich auch über die Schaltfläche ⇄ Nix-Brick verschieben in einen anderen Host oder ein anderes Modul übertragen.

1.3Dateitypen

Jede Datei im Config-Verzeichnis wird einem Typ zugewiesen, der bestimmt, welches Panel angezeigt wird:

  • NixOS-Modul – configuration.nix und eingebundene Module; formularbasiert mit Nix-Brix
  • Home Manager – Dateien im HM-Verzeichnis (Standard: home/); eigene Felder pro Benutzer
  • Flake – flake.nix mit Hosts, Channel-Auswahl und Architektur
  • Hardware-Konfiguration – hardware-configuration.nix; schreibgeschützt (NixOS-generiert)
  • Flake Lock – flake.lock; nur Anzeige, keine Bearbeitung über NiCo
  • Nicht erkannte Datei – Raw-Modus: Direktbearbeitung im Codefenster, Änderungen werden sofort gespeichert

2Erste Schritte

2.1Erster Start

Beim ersten Start fragt NiCo nach dem Config-Verzeichnis. Wer noch keines hat, legt dort ein neues an; wer eine bestehende Konfiguration hat, wählt das vorhandene Verzeichnis.

1. Config-Verzeichnis angeben → NiCo richtet automatisch ein Git-Repository ein und kopiert hardware-configuration.nix aus /etc/nixos (falls vorhanden) 2. Bei gefundener oder fehlender Config: Import-Angebot (siehe 2.2) 3. Optional: Symlink /etc/nixos → Config-Verzeichnis einrichten (damit nixos-rebuild die Konfiguration automatisch findet)

NiCo speichert seine eigenen Einstellungen in ~/.config/nico/settings.json (XDG) – nie im Config-Verzeichnis selbst. Konfigurationseinstellungen wie Verzeichnisnamen für Hosts, Module und Home Manager liegen in {config-dir}/config.json.

2.2Config importieren

Findet NiCo eine bestehende NixOS-Konfiguration im gewählten Verzeichnis, bietet es den Import an. Erkannte Felder werden als Formularwerte übernommen; unbekannte Abschnitte landen als Nix-Brick 1-imported-config und bleiben vollständig erhalten.

  • Vor dem Import wird automatisch ein Sicherungspunkt angelegt
  • hardware-configuration.nix wird kopiert, falls vorhanden
  • Import ist auch nachträglich aufrufbar – aus einem anderen Verzeichnis oder einer ZIP-Datei
  • Ist das Zielverzeichnis nicht leer, wird vor dem Import ein Backup angelegt

3Arbeiten mit NiCo

3.1Oberfläche

  • Links – Dateibaum: alle .nix-Dateien im Config-Verzeichnis; Klick öffnet das jeweilige Panel
  • Rechts – Panel: Formularfelder für bekannte Optionen der ausgewählten Datei, darunter die Nix-Brix
  • Admin-Bereich: Einstellungen, Git-Status, Rebuild-Optionen, Sprache und Theme

3.2Vorschau vs. echter Dateistand

Die Schaltfläche Vorschau zeigt, welche .nix-Datei beim nächsten Schreiben erzeugt würde – ohne die Datei auf Platte zu verändern. Dry-Run und Rebuild arbeiten immer mit dem tatsächlichen Dateistand auf Platte, nicht mit dem Formularinhalt.

Merksatz: Formular ändern → Schreiben → Dry-Run/Rebuild. Ohne Schreiben prüft der Dry-Run den alten Stand.

3.3Schreiben, Dry-Run, Rebuild

1. Schreiben → Formularinhalt wird in die .nix-Dateien auf Platte geschrieben 2. Dry-Run → prüft die .nix-Dateien auf Syntaxfehler (nixos-rebuild dry-build) 3. Rebuild → baut das System (nixos-rebuild switch) – Live-Ausgabe direkt sichtbar

Bei aktiviertem Git legt NiCo vor jedem Rebuild automatisch einen Sicherungspunkt-Commit an – ein Rollback ist damit immer möglich. Im Flake-Modus wird der Rebuild für den aktiven Host ausgeführt.

Optional: Flake vor Rebuild aktualisieren (nix flake update). Diese Einstellung ist im Admin-Bereich konfigurierbar.

4Flakes, Hosts und Home Manager

Flake-Modus

Im Flake-Modus verwaltet NiCo eine flake.nix mit einem oder mehreren Hosts in nixosConfigurations. Das Flake-Panel bietet:

  • Channel-Auswahl: NiCo lädt beim Öffnen die aktuelle nixpkgs-Channelliste (nix-channels.s3.amazonaws.com, rein lesend). Ohne Internetzugang wird eine eingebaute Fallback-Liste verwendet
  • Architektur-Selektor: z. B. x86_64-linux, aarch64-linux
  • Hosts anlegen und löschen über die Oberfläche

Hosts

Jeder Host liegt in einem eigenen Verzeichnis (Standard: hosts/, konfigurierbar). NiCo erkennt alle Hosts aus nixosConfigurations in der flake.nix und stellt sie im Dateibaum dar. Jeder Host hat seine eigene configuration.nix mit eigenen Formularfeldern und Nix-Brix.

Home Manager

NiCo verwaltet Home Manager-Konfigurationen in einem eigenen Verzeichnis (Standard: home/, konfigurierbar). Pro Benutzer gibt es eine eigene .nix-Datei mit eigenen Formularfeldern – darunter home-manager.enable und weitere HM-Optionen. Nix-Brix sind auch in HM-Dateien möglich.

Btrfs (Auto-Scrub und Auto-Balance)

NiCo kann systemd-Timer und -Services für regelmäßige Btrfs-Wartung generieren:

  • Auto-Scrub: prüft regelmäßig die Integrität des Btrfs-Dateisystems
  • Auto-Balance: gleicht die Datenverteilung aus; konfigurierbar mit Häufigkeit, dusage- und musage-Schwellwerten

5Git-Integration

5.1Git-Guard und Sicherungspunkte

Der Git-Guard prüft vor jedem Schreibvorgang den Git-Status des Config-Verzeichnisses. Uncommittete externe Änderungen werden gemeldet und müssen bestätigt werden, bevor NiCo schreibt – so gehen keine Änderungen unbemerkt verloren.

Vor jedem Rebuild legt NiCo automatisch einen Sicherungspunkt-Commit an. Damit ist der Zustand vor dem Rebuild immer als Git-Commit gesichert, auf den jederzeit zurückgerollt werden kann.

5.2Commits, Rollback, Push

  • Auto-Commit: nach jedem Schreiben automatisch committen (konfigurierbar)
  • Manueller Commit: über den Git-Bereich im Admin-Panel mit eigener Nachricht
  • Rollback: setzt das Config-Verzeichnis auf den letzten Commit zurück; ungespeicherte Änderungen gehen dabei verloren
  • Push: manuell oder als Auto-Push nach jedem Commit

5.3Remote-Repository

Eine Remote-URL wird in den Admin-Einstellungen eingetragen. NiCo zeigt dort an, ob der lokale Stand ahead oder behind dem Remote ist. Auto-Push ist optional aktivierbar – dann wird nach jedem Commit automatisch gepusht.

5.4Fremddateien

Liegen im Config-Verzeichnis Dateien, die nicht zur NixOS-Konfiguration gehören (z. B. Scripts, README, Backups), fragt NiCo vor dem Sicherungspunkt-Commit pro Datei, ob sie in Git aufgenommen oder ausgenommen (auf die .gitignore gesetzt) werden soll. Die Abfrage lässt sich in den Admin-Einstellungen abschalten.

Für bereits in Git erfasste Fremddateien gibt es im Admin-Bereich einen Aufräum-Dialog: Austragen entfernt die Datei aus Git und setzt sie auf die .gitignore (die Datei bleibt auf der Platte, die Git-Historie bleibt erhalten); Behalten merkt sich die Datei dauerhaft als gewollt.

6Import, Export und Entkoppeln

Diese Funktionen sind über den Admin-Bereich erreichbar.

Import

Eine bestehende NixOS-Konfiguration aus einem Verzeichnis oder einer ZIP-Datei einlesen. Erkannte Felder werden übernommen, unbekannte Abschnitte als Nix-Brick gesichert. Ist das Zielverzeichnis nicht leer, wird vor dem Import ein Backup angelegt.

Export

Die gesamte Konfiguration als ZIP-Datei herunterladen – alle .nix-Dateien aus dem Config-Verzeichnis werden eingepackt. Nützlich als Backup oder zum Übertragen auf ein anderes System.

Entkoppeln

NiCo löst die Verbindung zum Config-Verzeichnis. Vorher wird automatisch ein ZIP-Backup angelegt. Dann entfernt NiCo seine Marker-Kommentare aus den .nix-Dateien (der Nix-Inhalt bleibt vollständig erhalten) und löscht config.json bzw. eine alte nico.json. Das Verzeichnis ist danach eine reine Nix-Konfiguration ohne NiCo-Spuren und kann manuell oder mit einem anderen Tool weiterverwendet werden.

7Typische Fehler und Grenzen

Dry-Run oder Rebuild schlägt sofort fehl

Wahrscheinlichste Ursache: Formularinhalt wurde noch nicht geschrieben. Dry-Run und Rebuild arbeiten immer mit dem Stand auf Platte – zuerst Schreiben, dann prüfen oder bauen.

Nix-Fehler im Dry-Run/Rebuild

Die vollständige Ausgabe von Dry-Run und Rebuild ist direkt im Stream sichtbar. Nix-Fehlermeldungen können dort gelesen werden. Häufige Ursachen sind Syntaxfehler in Nix-Brix oder Referenzen auf nicht existierende Pakete.

NiCo erkennt eine .nix-Konstruktion nicht

Komplexe oder ungewöhnliche Nix-Syntax, die NiCo nicht parsen kann, wird niemals gelöscht. Solche Abschnitte bleiben als Nix-Brick erhalten. Bei einem Import landen sie automatisch im Brick 1-imported-config.

hardware-configuration.nix kann nicht bearbeitet werden

Diese Datei ist in NiCo schreibgeschützt, weil sie von NixOS automatisch generiert wird. Manuelle Änderungen daran sind zwar technisch möglich, können aber beim nächsten nixos-generate-config überschrieben werden.

Snapper-Validator meldet Fehler

Der Validator prüft, ob konfigurierte Snapper-Mountpoints existieren und tatsächlich btrfs-Dateisysteme sind. Ist ein Mountpoint nicht gemountet oder kein btrfs, wird das als Fehler gemeldet. Das ist kein NiCo-Fehler, sondern ein Hinweis auf eine inkonsistente Systemkonfiguration.

Validator-Meldung: host_orphaned

Ein Host-Verzeichnis existiert, ist aber nicht in nixosConfigurations der flake.nix eingebunden. Das kann Absicht sein (z. B. ein Host im Aufbau). NiCo meldet es als Hinweis, nicht als Fehler.

Grenzen von NiCo

  • NiCo ist kein Paket-Browser und keine Suchmaschine für Nix-Optionen
  • NiCo generiert keine Konfiguration aus dem Nichts – es arbeitet immer mit bestehenden oder manuell erstellten .nix-Dateien
  • NiCo ist kein Ersatz für das Lesen der NixOS-Dokumentation oder des NixOS-Wikis
  • NiCo läuft nicht auf mehreren Rechnern gleichzeitig – es ist ein lokales Werkzeug für eine Maschine
